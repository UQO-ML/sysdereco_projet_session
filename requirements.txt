# ---- ECR experiment runtime ---------------------------------------------
# Base : image `nvcr.io/nvidia/pytorch:26.03-py3`. Elle fournit deja :
#   torch, torchvision, torchaudio, numpy, pandas, scipy, scikit-learn, tqdm,
#   matplotlib, PyYAML, jupyterlab>=4, ipykernel, regex, tokenizers, datasets,
#   safetensors, huggingface-hub, requests.
# On ne les re-liste pas ici : (1) pip echoue a desinstaller les paquets apt
# (erreur `uninstall-no-record-file` pour wheel/PyYAML/pygments), (2) on evite
# un re-download inutile, (3) on s'evite les downgrades (torch NVIDIA 2.11.0a).
#
# Article : python==3.8 / torch==1.8.1 / transformers==4.15 / accelerate==0.8 /
# pyg==2.0.1. On cible la stack moderne compatible : transformers 4.x,
# accelerate >= 0.28, peft >= 0.10, torch-geometric >= 2.5.
#
# NB : l'installation est faite avec `--no-build-isolation` dans le Dockerfile
# (sinon flash-attn echoue a builder : son setup.py fait `import torch` qui
# n'est pas visible depuis un build env isole). Apres installation, le
# Dockerfile fige torch / torchvision / torchaudio / flash-attn / vllm /
# torch-geometric dans `/etc/pip/constraint.txt` (variable NGC `PIP_CONSTRAINT`).
# Tout `pip install <paquet>` ulterieur dans le conteneur refusera alors
# d'up/downgrader ces paquets ; pip signalera un conflit explicite plutot
# que de remplacer silencieusement ~2-3 GB de torch+CUDA wheels.
#
# Note importante : ce n'est PAS `torch-geometric` qui force le downgrade de
# torch, contrairement a une intuition courante. Le vrai coupable est
# `vllm 0.19.x`, dont les pins stricts `torchaudio==2.10.0` et
# `torchvision==0.25.0` entrainent transitivement `torch==2.10.0` (donc
# downgrade depuis la build NGC torch 2.11.0a0). torch-geometric 2.7, lui,
# ne declare aucun pin torch (il s'aligne sur n'importe quelle version
# installee).
